# Product Brief Addendum

## Feature Name

Drag-and-Drop Route Reordering

## Feature Type

New MVP Enhancement

## One-Sentence Description

Allow users to drag active route locations to a new position in the route list so they can quickly reorder stops without clicking up or down one step at a time.

---

## 1. Feature Overview

### Problem

The current route ordering experience relies on moving locations up or down one position at a time. This works for small lists, but it becomes frustrating when a user wants to move a stop across several positions, such as moving the last stop to the beginning or placing a location in the middle of the route.

### Solution

Add drag-and-drop sorting to the active route list. Users should be able to grab a location card, drag it to a new position, and drop it anywhere in the active route order. Once dropped, the app should update the route order, map markers, walking route, walking times, total walking time, and total distance.

### Primary User

A user planning a walking route across several saved locations who wants to quickly test different route sequences.

### Why This Feature Should Exist

Route planning is an iterative activity. Users often need to compare several possible stop orders before choosing the best walking route. Drag-and-drop makes route editing faster and more intuitive than repeatedly clicking “Move earlier” or “Move later.”

---

## 2. Goals and Non-Goals

### Product Goals

This feature should help users:

* Quickly move a location to any position in the active route list.
* Compare route sequences with less manual clicking.
* Understand that dragging changes route order only, not location details.
* See the map route and walking times update after the new order is confirmed.
* Use fallback controls if drag-and-drop is difficult on their device.

### Business Goals

This feature should help the creator:

* Improve the perceived polish and usability of the route planner.
* Make the app feel more like a modern interactive planning tool.
* Reduce friction for users testing different walking route options.

### Non-Goals

This feature will not include:

* Editing location names, addresses, coordinates, or saved details.
* Dragging locations between multiple saved route plans.
* Dragging locations onto the map to change their geographic position.
* Automatic route optimization.
* Multi-user collaborative dragging.
* Complex animations beyond basic drag feedback.

---

## 3. Users and Use Cases

### Primary User

#### User Type

Walking route planner

#### Description

A person organizing a walking route across multiple saved locations who wants to quickly adjust the order of stops.

#### Goals

* Move a stop several positions at once.
* Try different route sequences quickly.
* See the route update after reordering.
* Preserve all original location details while changing only route order.

#### Pain Points

* Moving a stop one position at a time is slow.
* Large route lists become tedious to reorganize.
* Users may worry that changing order could accidentally edit or delete location details.
* Mobile users need controls that still work if dragging is difficult.

### Core Use Cases

* Drag a stop from the bottom of the route to the top.
* Drag a stop from the middle of the route to a different position.
* Drop a stop into a new order and see the route recalculate.
* Use up/down buttons as a fallback if drag-and-drop is not convenient.

### Jobs To Be Done

* When I am planning a route with many stops, I want to drag a location directly to a new position, so I can change the route faster.
* When I change the order of my stops, I want the map and walking times to update, so I can compare route options.
* When I reorder stops, I want the app to preserve the location details, so I can safely experiment with the route sequence.

## 5. MVP Scope

### Feature Summary

This feature adds a faster route reordering interaction on top of the existing MVP. Users can drag active route location cards into a new order, and the app updates the route after the drop.

### Must Have

These items are required for this feature:

* Drag-and-drop sorting for active route locations.
* Visible drag handle on each active location card.
* Reordering updates `activeLocationIds`.
* Reordering does not change location name, address, coordinates, or saved details.
* Route recalculation after drop.
* Walking time recalculation after drop.
* Total walking time recalculation after drop.
* Total distance recalculation after drop.
* Map marker number updates after drop.
* Visual dragging state.
* Visible drop target or insertion position.
* Up/down buttons retained as fallback controls.
* Desktop and mobile-friendly interaction.

### Out of Scope

Do not build these as part of this feature:

* Editing location details.
* Dragging locations on the map.
* Dragging removed locations into the active route list.
* Dragging active locations into the removed list.
* Saving multiple route versions.
* Automatic route optimization.
* Collaborative route editing.
* Advanced animation-heavy interactions.

## 6. User Flows

### Flow 1: Drag a Location to a New Route Position

1. User views the active route list.
2. User sees numbered location cards with visible drag handles.
3. User grabs the drag handle on a location card.
4. App visually lifts or highlights the selected card.
5. User drags the card to a new position in the active route list.
6. App shows a clear drop position.
7. User releases the card.
8. App updates the active route order.
9. App updates route numbering.
10. App recalculates the walking route, walking times, total walking time, and total distance.
11. App updates the map route and marker numbers.

### Flow 2: Dragging Without Changing Position

1. User grabs a location card.
2. User drags the card but drops it back in the same position.
3. App keeps the active route order unchanged.
4. App does not unnecessarily recalculate route data.
5. User remains on the active route list.

### Flow 3: Reorder Using Fallback Buttons

1. User views the active route list.
2. User chooses “Move earlier” or “Move later” on a location card.
3. App changes the route order by one position.
4. App updates route numbering.
5. App recalculates the route, walking times, total walking time, and total distance.
6. App updates the map route and marker numbers.

### Flow 4: Mobile Drag Reorder

1. User opens the app on a phone-sized screen.
2. User sees the active route list above the map.
3. User long-presses or grabs the drag handle on a location card.
4. User drags the card to a new position.
5. User releases the card.
6. App updates the route order and recalculates route details.
7. User can still use fallback move buttons if dragging is difficult.

## 7. Data Model Impact

### Data Storage Choice

For this feature, continue to use:

* [x] Local storage
* [ ] Replit database
* [ ] Supabase
* [ ] Firebase
* [ ] Postgres
* [ ] Other: Not needed for this feature

Recommended choice:
Continue using local storage. This feature only changes the order of existing active locations, so it does not require a new database or backend.

### Impacted Entity: RoutePlan

Description:
The route plan already stores the ordered list of active locations. Drag-and-drop reordering should update the order of this array.

| Field               | Type     | Required? | Notes                                                                                             |
| ------------------- | -------- | --------- | ------------------------------------------------------------------------------------------------- |
| activeLocationIds   | string[] | Yes       | This is the source of truth for active route order. Drag-and-drop should reorder this array only. |
| totalWalkingMinutes | number   | No        | Recalculated after route order changes.                                                           |
| totalDistanceMeters | number   | No        | Recalculated after route order changes.                                                           |
| updatedAt           | datetime | Yes       | Updated after successful reorder.                                                                 |

### Impacted Entity: Location

Description:
Location objects should not be modified when users drag and drop route items.

| Field     | Type   | Required? | Notes                                                                              |
| --------- | ------ | --------- | ---------------------------------------------------------------------------------- |
| id        | string | Yes       | Used to reorder `activeLocationIds`.                                               |
| name      | string | Yes       | Must remain unchanged during reorder.                                              |
| address   | string | Yes       | Must remain unchanged during reorder.                                              |
| latitude  | number | No        | Must remain unchanged during reorder.                                              |
| longitude | number | No        | Must remain unchanged during reorder.                                              |
| status    | string | Yes       | Must remain unchanged unless the user explicitly removes or restores the location. |

### Relationships

| Entity    | Relationship        | Related Entity           |
| --------- | ------------------- | ------------------------ |
| RoutePlan | has ordered list of | Location                 |
| RoutePlan | has many            | RouteLeg                 |
| RouteLeg  | belongs to          | Location as fromLocation |
| RouteLeg  | belongs to          | Location as toLocation   |

### Validation Rules

* Drag-and-drop reordering must only affect active locations.
* Removed locations must not appear in the active sortable list.
* A dropped item must result in a valid `activeLocationIds` array.
* `activeLocationIds` must contain unique location IDs.
* Reordering must not create duplicate location IDs.
* Reordering must not delete location IDs.
* Reordering must not modify the underlying location object.
* If the item is dropped in the same position, no route recalculation is required.

### Example Data Change

Before drag:

```json
{
  "activeLocationIds": ["loc-1", "loc-2", "loc-3", "loc-4"]
}
```

User drags `loc-4` to the second position.

After drag:

```json
{
  "activeLocationIds": ["loc-1", "loc-4", "loc-2", "loc-3"]
}
```

The `Location` objects remain unchanged.

## 9. Functional Requirements

### Drag-and-Drop Reordering

* User can drag active location cards to a new position.
* User can move a location multiple positions in one action.
* User can drop a location at the beginning, middle, or end of the active route list.
* App updates `activeLocationIds` after the drop.
* App does not modify the underlying location object.
* App recalculates route data after a successful drop.
* App does not recalculate route data continuously while dragging.
* App does not recalculate if the item is dropped in the same position.

### Fallback Reordering

* App keeps “Move earlier” and “Move later” buttons.
* Buttons remain functional even if drag-and-drop fails.
* Buttons update the same `activeLocationIds` source of truth.
* Buttons trigger the same route recalculation behavior as drag-and-drop.

### Map and Route Updates

* Map marker numbering updates after reorder.
* Walking route updates after reorder.
* Walking time between stops updates after reorder.
* Total walking time updates after reorder.
* Total distance updates after reorder.

## 10. Non-Functional Requirements

### Usability

* The drag handle should be easy to discover.
* The user should understand that dragging changes route order only.
* The UI should provide clear visual feedback while dragging.
* The route should update only after the user drops the item.
* The interaction should feel fast and stable.

### Performance

* Do not call routing APIs continuously during drag.
* Recalculate route data only after a successful drop.
* Debounce or guard route recalculation if needed.
* The feature should work smoothly for up to 20 active locations.

### Reliability

* If drag-and-drop fails, fallback move buttons should still work.
* If route recalculation fails after a reorder, preserve the new order and show a route error message.
* If the app is refreshed after reorder, the new order should persist from local storage.

### Accessibility

* Keep keyboard-accessible fallback controls.
* Drag handles should have accessible labels.
* Move earlier and move later buttons should have clear accessible labels.
* Color should not be the only indicator of dragging or drop position.

## 11. Technical Approach

### Recommended Library

Use `dnd-kit` for drag-and-drop sorting.

Recommended packages:

```txt
@dnd-kit/core
@dnd-kit/sortable
@dnd-kit/utilities
```

### Why `dnd-kit`

* Good fit for React.
* Supports sortable lists.
* Works with pointer and touch interactions.
* Flexible enough for a custom card-based UI.
* Better suited for new projects than older drag-and-drop libraries.

### Implementation Notes

* Treat `activeLocationIds` as the source of truth for route order.
* Render active route cards from `activeLocationIds`.
* Wrap the active route list in a sortable context.
* Give each location card a stable `id`.
* On drag end:

  * Identify the old index and new index.
  * Reorder `activeLocationIds`.
  * Save updated route plan to local storage.
  * Recalculate route legs.
  * Recalculate walking times.
  * Recalculate total walking time.
  * Recalculate total distance.
  * Update marker numbering.
* Do not mutate the `locations` array or individual location objects.
* Do not call the routing API during drag movement.
* Keep up/down controls as fallback actions using the same reorder function.

### Suggested Component Updates

```txt
src/
├── components/
│   ├── RouteList.jsx
│   ├── SortableRouteItem.jsx
│   └── RouteSummary.jsx
├── utils/
│   └── reorderLocations.js
```

### Suggested Helper Function

```js
function reorderArray(items, oldIndex, newIndex) {
  const result = [...items];
  const [removed] = result.splice(oldIndex, 1);
  result.splice(newIndex, 0, removed);
  return result;
}
```

## 13. Acceptance Criteria

### Drag-and-Drop Behavior

* Given there are at least 2 active locations, when the user drags a location card to a new position, then the card appears in the new position after drop.
* Given the user drags a location from the bottom of the list to the top, when they drop it, then the location becomes the first stop.
* Given the user drags a location from the top of the list to the bottom, when they drop it, then the location becomes the last stop.
* Given the user drops a location in the same position, then the route order remains unchanged.

### Data Integrity

* Given a user reorders locations, then only `activeLocationIds` changes.
* Given a user reorders locations, then the location name remains unchanged.
* Given a user reorders locations, then the location address remains unchanged.
* Given a user reorders locations, then the location coordinates remain unchanged.
* Given a user reorders locations, then no active location IDs are duplicated or deleted.

### Route Recalculation

* Given the active route order changes after drop, then the walking route recalculates.
* Given the active route order changes after drop, then walking times between stops recalculate.
* Given the active route order changes after drop, then total walking time recalculates.
* Given the active route order changes after drop, then total distance recalculates.
* Given the active route order changes after drop, then map marker numbers update.

### Visual Feedback

* Given the user starts dragging a stop, then the dragged card shows a visual dragging state.
* Given the user drags a card between two stops, then the UI shows where the card will be dropped.
* Given the route is recalculating after drop, then the app shows an updating state if recalculation takes noticeable time.

### Fallback Controls

* Given drag-and-drop is available, then “Move earlier” and “Move later” buttons are still visible or accessible.
* Given the user clicks “Move earlier,” then the stop moves one position earlier.
* Given the user clicks “Move later,” then the stop moves one position later.
* Given the fallback buttons reorder the route, then route recalculation behaves the same as drag-and-drop.

### Persistence

* Given the user reorders the route, when they refresh the page, then the new route order is restored from local storage.
