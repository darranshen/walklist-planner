# Product Brief Addendum

## Feature Name

Automatic Route Optimization

## Feature Type

New Feature Update

## One-Sentence Description

When users import or load locations from a Google Maps shared list, the app should automatically suggest an optimized walking route that minimizes total walking distance and time across all active locations.

---

## 1. Feature Overview

### Problem

When a user imports a Google Maps shared list, the saved locations may appear in the same order as the original list, but that order is not necessarily the best walking route. Users may have to manually reorder many locations to reduce backtracking, walking distance, and total travel time.

### Solution

Add an automatic route optimization feature. After locations are imported or loaded, the app should calculate a suggested route order that minimizes total walking distance and estimated walking time. The optimized order should become the active route order, while still allowing users to manually adjust the order afterward using drag-and-drop or move controls.

### Primary User

A user planning a walking route from multiple saved Google Maps locations who wants the app to recommend an efficient route order automatically.

### Why This Feature Should Exist

The core value of WalkList Planner is helping users turn saved places into a practical walking route. Automatic optimization reduces manual planning effort and makes the app immediately useful after import.

---

## 2. Goals and Non-Goals

### Product Goals

This feature should help users:

* Automatically create a more efficient walking route after importing locations.
* Minimize total walking distance and estimated walking time.
* Reduce route backtracking.
* Understand that the optimized route is a suggestion they can still edit.
* Continue manually reordering stops after optimization if desired.

### Business Goals

This feature should help the creator:

* Make the app feel substantially more useful than a basic map/list interface.
* Improve first-run value after a Google Maps shared list is imported.
* Differentiate the app from manually editing directions in Google Maps.
* Create a stronger demo experience for route planning use cases.

### Non-Goals

This feature will not include:

* Guaranteed mathematically perfect optimization for very large lists.
* Real-time traffic-based walking optimization.
* Multi-modal route optimization across walking, transit, driving, and biking.
* Opening hours optimization.
* User preference optimization, such as scenic routes, food priority, or accessibility.
* Multiple route alternatives in the first version.
* Optimizing across multiple days.
* Collaborative optimization.

## 3. Users and Use Cases

### Primary User

#### User Type

Walking route planner

#### Description

A person importing several saved locations from Google Maps who wants a good route order without manually testing every possible sequence.

#### Goals

* Import or load multiple locations.
* Automatically get a sensible walking order.
* Reduce total distance and walking time.
* Avoid obvious backtracking.
* Still adjust the route manually afterward.

#### Pain Points

* Imported locations may be unordered or ordered by save time.
* Manually reordering many stops is tedious.
* It is hard to know which stop should come first, second, or last.
* Users may not know whether their route is efficient.

### Core Use Cases

* User imports a Google Maps shared list and receives an optimized route order.
* User clicks “Optimize Route” after manually adding or editing the active location set.
* User removes a location and re-optimizes the remaining route.
* User restores a location and re-optimizes the route.
* User accepts the optimized order but manually adjusts one or two stops afterward.

### Jobs To Be Done

* When I import a saved list of places, I want the app to suggest the best walking order, so I do not have to manually plan the route.
* When I have many stops, I want the app to reduce total walking distance, so I can spend less time walking between places.
* When the app changes my route order, I want to still be able to edit it, so I can account for my own preferences.

## 5. Feature Scope

### Feature Summary

This feature adds automatic route optimization after Google Maps list import and as an on-demand action. The app should suggest an efficient route order across all active locations, then update the map, walking route, walking times, total walking time, total distance, and marker numbering.

### Must Have

These items are required for this feature:

* Automatic optimization after imported locations load.
* Manual “Optimize Route” button.
* Optimization based on walking distance and estimated walking time.
* Updated active route order after optimization.
* Updated map marker numbering.
* Updated walking route.
* Updated walking times between stops.
* Updated total walking time.
* Updated total walking distance.
* Loading state while optimization runs.
* Success state after optimization completes.
* Error state if optimization fails.
* Preserve previous route order if optimization fails.
* Manual reorder still available after optimization.
* Location details remain unchanged.
* Practical optimization approach that works for MVP-sized lists.

### Out of Scope

Do not build these as part of this feature:

* Perfect mathematical optimization for very large lists.
* Multiple route alternatives.
* Automatic route optimization across multiple days.
* Opening hours-based optimization.
* Accessibility-based optimization.
* Scenic route optimization.
* Food/category preference optimization.
* Weather-aware optimization.
* Real-time collaborative optimization.
* Transit, biking, or driving optimization.
* Editing location details during optimization.

## 6. User Flows

### Flow 1: Auto-Optimize After Google Maps Import

1. User imports or loads a Google Maps shared list.
2. App extracts or receives the saved locations.
3. App validates that at least 2 active locations are available.
4. App shows a loading state: “Optimizing route…”
5. App calculates an optimized walking order.
6. App updates `activeLocationIds` with the optimized order.
7. App recalculates the walking route, walking times, total walking time, and total distance.
8. App updates the map route and marker numbers.
9. App shows a success message with the optimized route summary.
10. User can manually adjust the optimized route if desired.

### Flow 2: Manual Optimize Route

1. User views the active route list.
2. User clicks “Optimize Route.”
3. App checks that at least 2 active locations are available.
4. App shows “Optimizing route…”
5. App calculates the optimized order.
6. App updates the active route order.
7. App recalculates route details.
8. User sees the optimized route on the map.
9. User can keep the result or manually reorder stops.

### Flow 3: Optimization After Adding or Restoring Locations

1. User adds or restores a location.
2. App updates the active route list.
3. User clicks “Optimize Route.”
4. App calculates a new optimized order using all active locations.
5. App updates the route order and route details.
6. User sees the new optimized route.

### Flow 4: Optimization Failure

1. User clicks “Optimize Route” or imports a list.
2. App attempts to calculate the optimized route.
3. Optimization fails because routing data is unavailable or an API call fails.
4. App preserves the previous active route order.
5. App shows a friendly error message.
6. User can continue using the existing route order or manually reorder stops.

### Validation Rules

* Optimization requires at least 2 active locations.
* Optimization must only use active locations.
* Removed locations must not be included in optimization.
* Optimized `activeLocationIds` must contain the same active location IDs as before optimization.
* Optimized `activeLocationIds` must not duplicate location IDs.
* Optimized `activeLocationIds` must not delete active location IDs.
* Optimization must not modify location details.
* If optimization fails, preserve the previous `activeLocationIds`.
* If there are exactly 2 active locations, optimization may keep the order unchanged unless start/end behavior is configured.

## 9. Functional Requirements

### Automatic Optimization

* App automatically runs route optimization after a new Google Maps shared list is imported and locations are successfully loaded.
* App only auto-optimizes when there are at least 2 active locations.
* App shows a loading state while optimization runs.
* App updates active route order after successful optimization.
* App recalculates route details after successful optimization.
* App shows a success message after optimization completes.

### Manual Optimization

* User can click “Optimize Route” at any time when there are at least 2 active locations.
* Manual optimization uses the current set of active locations.
* Manual optimization excludes removed locations.
* Manual optimization updates route order after success.
* Manual optimization preserves location details.
* Manual optimization shows an error if it cannot complete.

### Optimization Logic

* For MVP-sized lists, use a practical heuristic rather than brute-force permutations.
* Recommended approach:

  * Use nearest-neighbor to generate an initial route.
  * Optionally apply a 2-opt improvement pass to reduce obvious backtracking.
  * Use walking distance/time from the routing API where practical.
  * Use straight-line distance as a fallback when routing API data is unavailable.
* Optimization should prioritize minimizing total walking distance.
* If walking time estimates are available, use them as a secondary optimization signal.
* If the app supports a fixed start or fixed end later, preserve those constraints. For this version, assume all active stops can be reordered.

### Route Recalculation

* After optimization, recalculate walking route.
* After optimization, recalculate walking times between stops.
* After optimization, recalculate total walking time.
* After optimization, recalculate total distance.
* After optimization, update marker numbering.
* Save optimized route state to local storage.

## 11. Technical Approach

### Recommended Optimization Approach

Use a practical route optimization heuristic suitable for an MVP.

Recommended first version:

1. Build a distance matrix for active locations.
2. Use walking route distance/time from API if available.
3. Fall back to straight-line distance if API data is unavailable.
4. Generate an initial route using nearest-neighbor.
5. Apply a 2-opt improvement pass to reduce backtracking.
6. Return the optimized ordered list of location IDs.
7. Recalculate final route details using the existing routing service.

### Why Not Brute Force

A brute-force route optimizer checks every possible route order. This becomes too slow as the number of locations grows. For example:

```txt
5 locations = 120 possible orders
8 locations = 40,320 possible orders
10 locations = 3,628,800 possible orders
```

For the MVP, use nearest-neighbor plus optional 2-opt because it is fast, practical, and good enough for walking route planning.

* Optimization should update `activeLocationIds`, not the `locations` data.
* Do not include removed locations in the optimizer.

* Preserve previous route order in memory before running optimization.
* If optimization fails, restore or keep the previous route order.
* Run route recalculation only after the optimized order is selected.
* Use clear loading state to prevent duplicate optimization calls.

## 13. Acceptance Criteria

### Auto-Optimization After Import

* Given a user imports a Google Maps shared list with at least 2 locations, when the locations finish loading, then the app automatically starts route optimization.
* Given optimization is running, then the app shows “Optimizing route…”
* Given optimization succeeds, then the active route list updates to the optimized order.
* Given optimization succeeds, then the map route, marker numbers, walking times, total walking time, and total distance update.
* Given optimization succeeds, then the app shows a route optimization success message.

### Manual Optimization

* Given there are at least 2 active locations, when the user clicks “Optimize Route,” then the app optimizes the current active route order.
* Given there are fewer than 2 active locations, then the “Optimize Route” button is disabled or shows a helpful message.
* Given the user clicks “Optimize Route” multiple times quickly, then the app should not run multiple simultaneous optimization jobs.

### Data Integrity

* Given optimization succeeds, then `activeLocationIds` contains the same location IDs as before, only in a different order.
* Given optimization succeeds, then no location IDs are duplicated.
* Given optimization succeeds, then no active location IDs are removed.
* Given optimization succeeds, then removed locations remain removed.
* Given optimization succeeds, then location name, address, coordinates, status, and place ID remain unchanged.

### Route Updates

* Given optimization changes the route order, then walking times between stops recalculate.
* Given optimization changes the route order, then total walking time recalculates.
* Given optimization changes the route order, then total distance recalculates.
* Given optimization changes the route order, then marker numbering updates to match the route list.
* Given optimization returns the same order, then the app still shows the current route summary clearly.

### Error Handling

* Given optimization fails, then the previous route order is preserved.
* Given optimization fails, then the app shows: “We could not optimize this route. Your current stop order was preserved.”
* Given routing API data is unavailable, then the app uses approximate fallback optimization if possible.
* Given fallback estimates are used, then the app labels the result as approximate.

### Manual Editing After Optimization

* Given the route has been optimized, when the user drags a location to a new position, then the app updates the route based on the user’s manual order.
* Given the route has been optimized, when the user clicks move earlier or move later, then the app updates the route based on the user’s manual order.
* Given the user manually changes the optimized route, then the app recalculates route details.

### Persistence

* Given optimization succeeds, when the user refreshes the page, then the optimized route order is restored from local storage.
* Given optimization fails, when the user refreshes the page, then the last valid route order is restored.

## 17. Replit Build Prompt

Add an automatic route optimization feature to the existing WalkList Planner React app.

  * Walking times between consecutive stops
  * Total walking time
  * Total distance
  * Map marker numbering

* Prevent multiple optimization jobs from running at the same time.
* Show a success message after optimization, such as: “Route optimized: 4.6 mi, 71 min. You can still drag stops to adjust the order.”
* If optimization fails, preserve the previous route order and show: “We could not optimize this route. Your current stop order was preserved.”
* Save the optimized route order to local storage.
* Keep drag-and-drop reorder and move earlier / move later controls available after optimization.

1. Build a distance matrix for active locations.
2. Use walking distance and time from the routing API when available.
3. Use straight-line distance as a fallback in mock mode.
4. Generate an initial route using nearest-neighbor.

6. Return the optimized ordered list of active location IDs.

Do not build fixed start/end selection, round-trip mode, multiple route alternatives, opening hours optimization, scenic route preferences, accessibility routing, multi-day planning, undo optimization, collaboration, or export to Google Maps as part of this feature.
